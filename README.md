# Simon Delaigue - CV

Online resume website hosted on GitHub Pages.

## Workflow

- `cv-check.yml`: validates PDF and static files on push to `main`, `dev`, `feat/*`, `release/*`, `fix/*`, and on PR targeting `main`, `dev`, `release/*`.
- `deploy.yml`: publishes to GitHub Pages on push to `main`.
- `release.yml`: creates a GitHub Release on tag `cv-*` and uploads the CV PDF.

## Versioning Model

- `VERSION`: version of the CV document itself (`cv-x.y.z`).
- `version.json`: metadata displayed by the HTML page.
- `cvVersion`: CV document version (from `VERSION`).
- `siteVersion`: page deployment version (generated by CI, `site-<run_number>`).
- `deployedAt`: deployment date (UTC).

This separation avoids confusion: recruiters can read the CV version clearly, while the site deployment version remains traceable for operations.

## Release Process

1. Update your CV document version (for example `CV_VERSION`) to `cv-x.y.z`, then export the PDF.
2. Update `VERSION` with the exact same value.
3. Open a PR from `feat/*` or `fix/*` (or `release/*`) to `dev` or `main` according to your flow.
4. Merge to `main` (check/deploy run automatically).
5. Create and push the `cv-x.y.z` tag from `main`.
6. Verify the GitHub Release and the attached PDF asset.

## Tag Convention

- Format: `cv-x.y.z` (example: `cv-0.2.1`).
- The tag must exactly match the content of `VERSION`.
- PDF expectation: the exported PDF must contain the exact `VERSION` value as visible text.
